0+64# Requirements Document

## Introduction

This document defines the requirements for a **MERN stack** (MongoDB, Express.js, React, Node.js)
re-implementation of the existing Flask-based Digital Hospital Queue Management System.

### What is being built

A standalone web application located in a new `mern-app/` folder at the workspace root. It will
replicate every feature of the Flask app and add improvements enabled by the React SPA model
(real-time UI updates without full-page reloads, JWT-based auth, component-driven UI). The Flask
app is **not modified** in any way.

### How it maps to the Flask app

| Flask feature | MERN equivalent |
|---|---|
| `flask-login` / `session` | JWT stored in `httpOnly` cookie + React auth context |
| Jinja2 server-rendered pages | React SPA with React Router v6 |
| PyMongo direct queries | Mongoose ODM on Express.js REST API |
| ReportLab PDF generation | `pdfkit` / `puppeteer` PDF endpoint on Node.js |
| `appointments_col` polling via `/api/patient-appointments` | Axios polling (same pattern) |
| `ratings_col` | Same MongoDB collection, same schema |
| `hospitals_col` disease search | REST endpoint + React search component |
| Auto-cleanup (24 h) | Scheduled job via `node-cron` on the Express server |

### Project structure (new folder only)

```
mern-app/
  backend/          ← Express.js + Mongoose API server (port 5000)
  frontend/         ← React (Vite) SPA (port 5173)
```

## Glossary

- **System**: The MERN Hospital Queue application as a whole.
- **API_Server**: The Express.js + Node.js backend that exposes REST endpoints.
- **React_App**: The React (Vite) single-page application served to the browser.
- **Auth_Service**: The module inside API_Server responsible for JWT issuance and verification.
- **Appointment_Service**: The module inside API_Server that manages appointment CRUD and token logic.
- **Prescription_Service**: The module inside API_Server that manages prescription creation and PDF generation.
- **Rating_Service**: The module inside API_Server that manages doctor ratings and reviews.
- **Billing_Service**: The module inside API_Server that manages medicine pricing and bill generation.
- **Ambulance_Service**: The module inside API_Server that manages ambulance requests from patients.
- **Cleanup_Job**: The scheduled background task inside API_Server that expires old appointments.
- **Patient**: A registered user with role `patient`.
- **Doctor**: A registered user with role `doctor`.
- **Staff**: A registered user with role `staff`.
- **Admin**: A registered user with role `admin`.
- **Token**: A sequential integer assigned to each appointment on a given date for a given doctor, used to track queue position.
- **History**: An appointment flagged as `history: true`, meaning it is archived and no longer active.
- **JWT**: JSON Web Token used for stateless authentication.
- **Offline Appointment**: An appointment booked by Staff on behalf of a walk-in patient.
- **Bill**: A document generated by Staff containing medicine costs for a completed prescription, sent to the patient.

## Requirements

### Requirement 1: Project Scaffolding

**User Story:** As a developer, I want a well-structured MERN project scaffold, so that the
backend and frontend are clearly separated and independently runnable.

#### Acceptance Criteria

1. THE System SHALL contain a `mern-app/backend/` directory with an Express.js application entry point (`server.js` or `index.js`).
2. THE System SHALL contain a `mern-app/frontend/` directory with a React application bootstrapped via Vite.
3. THE API_Server SHALL listen on port `5000` by default, configurable via a `.env` variable `PORT`.
4. THE React_App SHALL be configured to proxy API requests to `http://localhost:5000` during development.
5. THE System SHALL include a root-level `mern-app/package.json` with `start`, `dev`, and `install:all` scripts that orchestrate both backend and frontend.

---

### Requirement 2: Database Connection

**User Story:** As a developer, I want the MERN backend to connect to the same MongoDB instance
as the Flask app, so that existing data (patients, doctors, appointments, etc.) is shared.

#### Acceptance Criteria

1. THE API_Server SHALL connect to MongoDB using Mongoose with the connection string read from the environment variable `MONGO_URI`.
2. WHEN the MongoDB connection fails on startup, THE API_Server SHALL log the error and exit with a non-zero status code.
3. THE API_Server SHALL use the existing database name `digital_hospital_queue` and the same collection names: `patients`, `doctors`, `staff`, `admin`, `appointments`, `hospitals`, `prescriptions`, `ratings`.
4. THE API_Server SHALL define Mongoose schemas that match the field shapes used by the Flask app so that documents written by either app are readable by the other.

---

### Requirement 3: Authentication — Login, Logout, Register

**User Story:** As a user (Patient, Doctor, Staff, or Admin), I want to log in with my username
and password, so that I can access my role-specific dashboard.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/auth/login` with valid `username`, `password`, and `role` fields, THE Auth_Service SHALL return a signed JWT containing `username`, `role`, and `userId`.
2. WHEN a POST request is made to `/api/auth/login` with invalid credentials, THE Auth_Service SHALL return HTTP 401 with a JSON error message.
3. THE Auth_Service SHALL set the JWT as an `httpOnly` cookie named `token` with `sameSite: strict` and `secure: true` in production.
4. WHEN a POST request is made to `/api/auth/logout`, THE Auth_Service SHALL clear the `token` cookie and return HTTP 200.
5. WHEN a POST request is made to `/api/auth/register` with `name`, `username`, and `password`, THE Auth_Service SHALL create a new Patient document and return HTTP 201.
6. IF a registration request is made with a `username` that already exists, THEN THE Auth_Service SHALL return HTTP 409 with a descriptive error message.
7. THE Auth_Service SHALL hash all passwords using `bcrypt` with a minimum cost factor of 10 before storing them in MongoDB.
8. THE React_App SHALL provide a `/login` route rendering a login form with role selector (patient, doctor, staff, admin), username, and password fields.
9. THE React_App SHALL provide a `/register` route rendering a registration form for new patients.
10. WHEN login succeeds, THE React_App SHALL redirect the user to their role-specific dashboard route.

---

### Requirement 4: Route Protection and Role-Based Access

**User Story:** As a system operator, I want all dashboard routes to be protected, so that
unauthenticated or unauthorised users cannot access them.

#### Acceptance Criteria

1. THE API_Server SHALL expose an Express middleware `requireAuth` that validates the JWT cookie on every protected route.
2. WHEN a request reaches a protected route without a valid JWT, THE API_Server SHALL return HTTP 401.
3. THE API_Server SHALL expose an Express middleware `requireRole(role)` that checks the `role` claim in the JWT.
4. WHEN a request reaches a role-restricted route with a JWT whose role does not match, THE API_Server SHALL return HTTP 403.
5. THE React_App SHALL implement a `ProtectedRoute` component that redirects unauthenticated users to `/login`.
6. THE React_App SHALL implement role-based route guards so that a Patient cannot navigate to the Doctor dashboard URL, and vice versa.

### Requirement 5: Patient Dashboard

**User Story:** As a Patient, I want a dashboard that shows my active appointments, appointment
history, prescriptions, and allows me to book, cancel, and reschedule appointments.

#### Acceptance Criteria

1. WHEN a GET request is made to `/api/patients/:username/dashboard`, THE API_Server SHALL return the patient's active appointments, history appointments, and prescriptions in a single JSON response.
2. THE React_App SHALL render the patient dashboard at `/patient/:username` showing active appointments, history, and prescriptions in separate sections.
3. WHEN the patient dashboard loads, THE React_App SHALL fetch and display the list of available doctors with their current status (available, on_leave, on_break, offline).
4. THE React_App SHALL poll `/api/patients/:username/appointments` every 10 seconds to refresh appointment statuses without a full page reload.

### Requirement 6: Book Appointment

**User Story:** As a Patient, I want to book an appointment with a doctor on a chosen date and
time, so that I can receive medical care.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/appointments/book` with `username`, `doctor`, `date`, and `time`, THE Appointment_Service SHALL validate that the time is between 09:00 and 20:59 inclusive.
2. IF the requested time is outside 09:00–20:59, THEN THE Appointment_Service SHALL return HTTP 400 with the message "Appointments allowed only between 09:00 AM and 09:00 PM".
3. WHEN a booking request is made, THE Appointment_Service SHALL check the doctor's `status_override`, `leave_dates`, and `break_start`/`break_end` fields.
4. IF the doctor's `status_override` is `offline`, THEN THE Appointment_Service SHALL return HTTP 400 with a message indicating the doctor is unavailable.
5. IF the requested date is in the doctor's `leave_dates`, THEN THE Appointment_Service SHALL return HTTP 400 with a message indicating the doctor is on leave.
6. IF the requested date is today and the requested time falls within the doctor's break window, THEN THE Appointment_Service SHALL return HTTP 400 with a message indicating the break window.
7. WHEN a booking is valid, THE Appointment_Service SHALL assign a Token equal to the count of existing appointments for that doctor on that date plus one.
8. WHEN a booking is valid, THE Appointment_Service SHALL insert the appointment with status `Booked`, `history: false`, and `created_at` set to the current UTC time.
9. THE React_App SHALL display a booking form on the patient dashboard with doctor selector, date picker, and time picker.
10. WHEN a booking succeeds, THE React_App SHALL display a success notification and refresh the appointments list.

### Requirement 7: Cancel Appointment

**User Story:** As a Patient, I want to cancel an active appointment, so that the slot is freed
and my history is updated.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/appointments/:id/cancel`, THE Appointment_Service SHALL set the appointment's `status` to `Cancelled` and `history` to `true`.
2. IF the appointment does not exist, THEN THE Appointment_Service SHALL return HTTP 404.
3. THE React_App SHALL provide a Cancel button for each active appointment on the patient dashboard.
4. WHEN cancellation succeeds, THE React_App SHALL remove the appointment from the active list and display a confirmation notification.

---

### Requirement 8: Reschedule Appointment

**User Story:** As a Patient, I want to reschedule an existing appointment to a new date and time,
so that I can adjust my schedule without losing my place in the queue.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/appointments/:id/reschedule` with `date` and `time`, THE Appointment_Service SHALL validate the new time is between 09:00 and 20:59.
2. IF the new time is outside the allowed window, THEN THE Appointment_Service SHALL return HTTP 400.
3. WHEN rescheduling is valid, THE Appointment_Service SHALL create a new appointment document with status `Rescheduled` and mark the original appointment as `history: true`.
4. THE React_App SHALL provide a Reschedule button that opens a modal with date and time pickers.
5. WHEN rescheduling succeeds, THE React_App SHALL refresh the appointments list and display a confirmation notification.

---

### Requirement 9: Auto-Cleanup of Expired Appointments

**User Story:** As a system operator, I want appointments older than 24 hours to be automatically
archived, so that the active queue stays clean.

#### Acceptance Criteria

1. THE Cleanup_Job SHALL run every hour using `node-cron`.
2. WHEN the Cleanup_Job runs, THE Cleanup_Job SHALL find all appointments where `created_at` is more than 24 hours before the current UTC time and `history` is `false`.
3. WHEN expired appointments are found, THE Cleanup_Job SHALL set `history: true` on each of them.
4. THE Cleanup_Job SHALL log the number of appointments archived on each run.

**User Story:** As a Doctor, I want a dashboard showing today's appointments, queue statistics,
sent prescriptions, and my ratings, so that I can manage my workday efficiently.

#### Acceptance Criteria

1. WHEN a GET request is made to `/api/doctors/:username/dashboard`, THE API_Server SHALL return today's appointments, summary counts (total, pending, completed, unique patients), sent prescriptions, and ratings for that doctor.
2. THE React_App SHALL render the doctor dashboard at `/doctor/:username` with sections for today's appointments, prescription history, ratings, and leave/status management.
3. THE React_App SHALL poll `/api/doctors/:username/appointments` every 10 seconds to refresh the appointment list.
4. THE React_App SHALL display the doctor's average rating and total review count prominently on the dashboard.

### Requirement 11: Doctor — Advance Queue Token

**User Story:** As a Doctor, I want to advance the queue token to the next patient, so that
patients know when it is their turn.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/doctors/:username/next-token`, THE Appointment_Service SHALL increment the current token counter for that doctor.
2. IF there are no appointments today for the doctor, THEN THE Appointment_Service SHALL return HTTP 400 with the message "No appointments today".
3. THE Appointment_Service SHALL reset the token counter to zero when the first call is made after a 24-hour gap since the last advance.
4. THE React_App SHALL display the current token number on the doctor dashboard and provide a "Next Patient" button.

### Requirement 12: Doctor — Mark Appointment Complete or Cancelled

**User Story:** As a Doctor, I want to mark an appointment as Completed or Cancelled, so that
the queue and patient history are kept accurate.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/appointments/:id/complete`, THE Appointment_Service SHALL set `status` to `Completed` and `history` to `true`.
2. WHEN a POST request is made to `/api/appointments/:id/doctor-cancel`, THE Appointment_Service SHALL set `status` to `Cancelled` and `history` to `true`.
3. THE React_App SHALL provide Done and Cancel action buttons for each appointment row on the doctor dashboard.

### Requirement 13: Send Prescription

**User Story:** As a Doctor, I want to send a prescription to a patient by entering their
username, medicines, and notes, so that the patient can view and download it.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/prescriptions` with `doctor`, `doctor_username`, `patient_username`, `medicines`, and `notes`, THE Prescription_Service SHALL verify the patient exists.
2. IF the patient does not exist, THEN THE Prescription_Service SHALL return HTTP 404 with a descriptive error.
3. IF the `medicines` field is empty, THEN THE Prescription_Service SHALL return HTTP 400.
4. WHEN the prescription is valid, THE Prescription_Service SHALL insert a prescription document with `created_at` set to the current UTC time.
5. THE React_App SHALL provide a prescription form on the doctor dashboard with a patient username lookup field, medicines textarea, and notes textarea.
6. WHEN the doctor types a patient username, THE React_App SHALL call `/api/patients/search?username=<value>` and display the patient's name if found.

### Requirement 14: Download Prescription as PDF

**User Story:** As a Patient or Doctor, I want to download a prescription as a formatted PDF,
so that I have an offline record.

#### Acceptance Criteria

1. WHEN a GET request is made to `/api/prescriptions/:id/pdf`, THE Prescription_Service SHALL generate a PDF document containing the doctor name, patient name, date, medicines list, and notes.
2. THE Prescription_Service SHALL return the PDF with `Content-Type: application/pdf` and `Content-Disposition: attachment; filename="prescription_<patient>_<date>.pdf"`.
3. IF the prescription ID does not exist, THEN THE Prescription_Service SHALL return HTTP 404.
4. THE React_App SHALL render a Download PDF button for each prescription in the patient and doctor dashboards.

### Requirement 15: Doctor Ratings and Reviews

**User Story:** As a Patient, I want to submit a star rating and written review for a completed
appointment, so that other patients can make informed choices.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/ratings` with `appointment_id`, `rating` (1–5), and `review`, THE Rating_Service SHALL verify the appointment exists and has status `Completed`.
2. IF the appointment is not Completed, THEN THE Rating_Service SHALL return HTTP 400.
3. THE Rating_Service SHALL upsert a rating document keyed on `appointment_id` so that a patient can update but not duplicate their review.
4. WHEN a DELETE request is made to `/api/ratings/:appointment_id`, THE Rating_Service SHALL remove the rating document and clear the `rating` and `review` fields from the appointment.
5. WHEN a GET request is made to `/api/doctors/:name/ratings`, THE Rating_Service SHALL return the average rating, total review count, and list of individual reviews.
6. THE React_App SHALL display a star-rating widget and review text box for each completed appointment that has not yet been reviewed.
7. THE React_App SHALL display a Delete Review button for completed appointments that already have a review.

### Requirement 16: Doctor Leave and Status Management

**User Story:** As a Doctor, I want to manage my leave dates, break window, and online/offline
status, so that patients cannot book appointments when I am unavailable.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/doctors/:username/leave` with `action: add_leave` and a `leave_date`, THE API_Server SHALL add the date to the doctor's `leave_dates` array using `$addToSet`.
2. WHEN a POST request is made to `/api/doctors/:username/leave` with `action: remove_leave` and a `leave_date`, THE API_Server SHALL remove the date from `leave_dates`.
3. WHEN a POST request is made to `/api/doctors/:username/leave` with `action: set_break`, `break_start`, and `break_end`, THE API_Server SHALL update the doctor's break window.
4. WHEN a POST request is made to `/api/doctors/:username/leave` with `action: set_status` and `status_override` of `available` or `offline`, THE API_Server SHALL update the doctor's `status_override` field.
5. WHEN a POST request is made to `/api/doctors/:username/clear-past-leaves`, THE API_Server SHALL remove all leave dates earlier than today from the doctor's `leave_dates` array.
6. WHEN a GET request is made to `/api/doctors/:name/status`, THE API_Server SHALL return the doctor's current status (`available`, `on_leave`, `on_break`, or `offline`) and a human-readable message.
7. THE React_App SHALL provide a Leave & Status management panel on the doctor dashboard with controls for adding/removing leave dates, setting break times, and toggling online/offline status.

### Requirement 17: Hospital Search by Disease or Specialty

**User Story:** As a Patient, I want to search for a hospital by disease or specialty, so that
I can find the right facility for my condition.

#### Acceptance Criteria

1. WHEN a GET request is made to `/api/hospitals/search?disease=<term>`, THE API_Server SHALL perform a case-insensitive regex search on the `disease` field of the `hospitals` collection.
2. IF a matching hospital is found, THE API_Server SHALL return the hospital's `name`, `specialty`, and `distance_km`.
3. IF no hospital is found, THE API_Server SHALL return HTTP 404 with the message "No hospital found for '<term>'".
4. THE React_App SHALL provide a search bar on the patient dashboard that calls the hospital search endpoint and displays the result inline.

### Requirement 18: Staff Dashboard — Live Appointment View

**User Story:** As a Staff member, I want to see all patient appointments in real time as they
are booked, so that I can manage the queue without manual refresh.

#### Acceptance Criteria

1. WHEN a GET request is made to `/api/staff/dashboard`, THE API_Server SHALL return all appointments for today across all doctors, sorted by token number, with live data from the `appointments` collection.
2. THE React_App SHALL render the staff dashboard at `/staff` showing the full appointment list with patient name, doctor, time, token, and status.

4. WHEN a patient books a new appointment (online), THE staff dashboard SHALL reflect it within the next polling cycle without requiring a page refresh.
5. THE React_App SHALL visually highlight newly added appointments (e.g., a brief highlight animation) so staff can spot new bookings quickly.

### Requirement 23: Staff — Offline Appointment Booking

**User Story:** As a Staff member, I want to book an appointment on behalf of a walk-in patient,
so that patients without internet access can still be added to the queue.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/staff/appointments/book` with `patient_username`, `doctor`, `date`, `time`, and `booked_by: "staff"`, THE Appointment_Service SHALL apply the same validation rules as online booking (time window, doctor availability, leave, break).
2. WHEN the offline booking is valid, THE Appointment_Service SHALL insert the appointment with `source: "offline"` and `booked_by_staff: <staff_username>` fields in addition to the standard fields.
3. IF the patient username does not exist in the `patients` collection, THEN THE Appointment_Service SHALL return HTTP 404 with a message asking staff to register the patient first.
4. THE React_App SHALL provide an "Offline Booking" form on the staff dashboard with patient username lookup, doctor selector, date picker, and time picker.
5. WHEN the patient username is entered, THE React_App SHALL call `/api/patients/search?username=<value>` and display the patient's full name for confirmation before submitting.
6. WHEN the offline booking succeeds, THE React_App SHALL display a success notification and the new appointment SHALL appear in the live appointment list immediately.

### Requirement 24: Staff — Ambulance Request Visibility

**User Story:** As a Staff member, I want to see ambulance requests submitted by patients in
real time, so that I can dispatch help promptly.

#### Acceptance Criteria

1. THE System SHALL add an `ambulance_requests` collection to MongoDB with fields: `patient_username`, `patient_name`, `location`, `reason`, `status` (`pending` / `dispatched` / `resolved`), and `created_at`.
2. WHEN a POST request is made to `/api/ambulance/request` with `patient_username`, `location`, and `reason`, THE Ambulance_Service SHALL insert a new ambulance request document with `status: "pending"`.
3. WHEN a GET request is made to `/api/staff/ambulance-requests`, THE API_Server SHALL return all ambulance requests with `status: "pending"` or `status: "dispatched"`, sorted by `created_at` descending.
4. THE React_App SHALL display an Ambulance Requests panel on the staff dashboard, polling `/api/staff/ambulance-requests` every 10 seconds.
5. WHEN a new ambulance request arrives, THE React_App SHALL show a prominent alert/badge on the staff dashboard so staff are immediately aware.
6. THE React_App SHALL allow staff to update the status of an ambulance request to `dispatched` or `resolved` via action buttons.
7. WHEN a PATCH request is made to `/api/ambulance/requests/:id/status` with a new `status`, THE Ambulance_Service SHALL update the request document accordingly.
8. THE React_App SHALL provide patients with an "Request Ambulance" button on the patient dashboard that opens a form with location and reason fields.

### Requirement 25: Staff — Doctor Activity Status Monitor

### Requirement 26: Staff — Billing After Prescription

**User Story:** As a Staff member, I want to receive prescriptions sent by doctors, set the
price for each medicine, and send the final bill to the patient, so that billing is handled
centrally by staff.

#### Acceptance Criteria

1. WHEN a doctor sends a prescription via `/api/prescriptions`, THE Prescription_Service SHALL set a `billing_status` field to `"pending_pricing"` on the prescription document.
2. WHEN a GET request is made to `/api/staff/prescriptions/pending-billing`, THE API_Server SHALL return all prescriptions where `billing_status` is `"pending_pricing"`, including the medicine list and patient details.
3. THE React_App SHALL display a Pending Billing panel on the staff dashboard listing all prescriptions awaiting pricing.
4. WHEN staff opens a prescription for billing, THE React_App SHALL render an editable table with one row per medicine, allowing staff to enter the unit price and quantity for each item.
5. WHEN a POST request is made to `/api/billing` with `prescription_id`, `staff_username`, and `line_items` (array of `{ medicine, quantity, unit_price }`), THE Billing_Service SHALL compute the `total_amount` as the sum of `quantity × unit_price` for all line items.
6. WHEN the bill is created, THE Billing_Service SHALL insert a document into a `bills` collection with fields: `prescription_id`, `patient_username`, `doctor`, `staff_username`, `line_items`, `total_amount`, `status: "sent"`, and `created_at`.
7. WHEN the bill is created, THE Billing_Service SHALL update the prescription's `billing_status` to `"billed"`.
8. WHEN a GET request is made to `/api/patients/:username/bills`, THE API_Server SHALL return all bills for that patient sorted by `created_at` descending.
9. THE React_App SHALL display a Bills section on the patient dashboard showing each bill's date, doctor, medicines, total amount, and a Download PDF button.
10. WHEN a GET request is made to `/api/billing/:id/pdf`, THE Billing_Service SHALL generate and return a formatted PDF bill with hospital name, patient name, doctor name, itemised medicine list with prices, and total amount.
11. IF a bill PDF is requested for a non-existent bill ID, THEN THE Billing_Service SHALL return HTTP 404.

### Requirement 19: Admin Dashboard

**User Story:** As an Admin, I want a dashboard to view all doctors and staff, add new doctors

#### Acceptance Criteria

4. WHEN a DELETE request is made to `/api/admin/doctors/:id`, THE API_Server SHALL remove the doctor document.
5. WHEN a POST request is made to `/api/admin/staff` with `name`, `username`, `password`, and `role`, THE API_Server SHALL create a new staff document.
6. IF the staff `username` already exists, THEN THE API_Server SHALL return HTTP 409.
7. WHEN a DELETE request is made to `/api/admin/staff/:id`, THE API_Server SHALL remove the staff document.
8. THE React_App SHALL render the admin dashboard at `/admin` with tabs or sections for Doctors, Staff, and Today's Appointments.

### Requirement 20: Live Appointment Status Polling

**User Story:** As a Patient or Doctor, I want appointment statuses to update automatically
without refreshing the page, so that I always see the current queue state.

#### Acceptance Criteria

1. THE React_App SHALL poll the patient appointments endpoint (`/api/patients/:username/appointments`) every 10 seconds while the patient dashboard is mounted.
2. THE React_App SHALL poll the doctor appointments endpoint (`/api/doctors/:username/appointments`) every 10 seconds while the doctor dashboard is mounted.
3. WHEN a polled response differs from the current UI state, THE React_App SHALL update only the changed appointment rows without re-rendering the entire page.
4. WHEN the component unmounts, THE React_App SHALL clear all polling intervals to prevent memory leaks.

### Requirement 21: Input Validation and Error Handling

**User Story:** As a system operator, I want all API inputs to be validated and errors to be
returned in a consistent format, so that the frontend can display meaningful messages.

#### Acceptance Criteria

1. THE API_Server SHALL validate all request bodies using a schema validation library (e.g., `express-validator` or `joi`).
2. WHEN validation fails, THE API_Server SHALL return HTTP 400 with a JSON body of the shape `{ "error": "<message>" }`.
3. THE API_Server SHALL include a global error-handling middleware that catches unhandled exceptions and returns HTTP 500 with `{ "error": "Internal server error" }` without leaking stack traces in production.
4. THE React_App SHALL display inline error messages next to form fields when the API returns a 400 response.
5. THE React_App SHALL display a toast or alert notification for 4xx and 5xx API errors that are not field-specific.

### Requirement 22: Security Baseline

**User Story:** As a system operator, I want the MERN app to follow basic security practices,
so that patient data is protected.

#### Acceptance Criteria

1. THE API_Server SHALL use the `helmet` middleware to set secure HTTP headers on all responses.
2. THE API_Server SHALL use the `cors` middleware configured to allow requests only from the React_App origin (`http://localhost:5173` in development).
3. THE Auth_Service SHALL sign JWTs with a secret read from the environment variable `JWT_SECRET` and set token expiry to 24 hours.
4. THE API_Server SHALL never return password fields in any API response.
5. THE API_Server SHALL use parameterised Mongoose queries and avoid constructing raw MongoDB query strings from user input.
